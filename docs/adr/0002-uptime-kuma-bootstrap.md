# 0002. Uptime Kuma Bootstrap

Date: 2026-06-07

## Status

Accepted

## Context

Ops Board should be runnable repeatedly during development without manually clicking through Uptime Kuma database setup, admin setup, monitor creation, and status page setup. Uptime Kuma v2 supports database setup through environment variables, but the current app source does not expose an official admin-user bootstrap environment variable.

The running Uptime Kuma v2.3.2 source confirms the relevant application contract:

- `UPTIME_KUMA_DB_TYPE` skips the database wizard.
- `UPTIME_KUMA_ENABLE_EMBEDDED_MARIADB=1` enables embedded MariaDB.
- Socket.IO `setup` creates the first admin user.
- Socket.IO `add` creates monitors.
- Socket.IO `addStatusPage`, `getStatusPage`, and `saveStatusPage` manage status pages.

## Decision

Ops Board will use Compose environment variables for Uptime Kuma database selection and a repo-owned bootstrap script for user, monitor, and status-page setup.

The bootstrap script will call Uptime Kuma's Socket.IO application contract. It will not seed Uptime Kuma tables directly.

The local admin password will live in `secrets/uptime_kuma_admin_password`, generated by `scripts/init-local-config.ps1`, and will not be committed or printed.

## Consequences

- Normal local setup becomes `init-local-config`, `docker compose up`, then `bootstrap-uptime-kuma`.
- Uptime Kuma monitor definitions move from prose-only docs into `stacks/uptime-kuma/bootstrap/monitors.yaml`.
- The bootstrap must be idempotent so it can run safely after restarts.
- If Uptime Kuma changes its Socket.IO contract in a future version, the bootstrap can fail and the manual UI remains the fallback.

## Guardrails

- Do not commit demo or production Uptime Kuma passwords.
- Do not use direct SQL writes for first-run setup.
- Keep the status page slug `ops-board`.
- Leave extra user-created monitors untouched.
